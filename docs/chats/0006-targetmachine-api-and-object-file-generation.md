# Chat 0006: TargetMachine API Migration and Direct Object File Generation

**Date**: 2025-12-10
**Model**: Cursor Composer 1
**Context**: Completing LLVM C API migration by adding TargetMachine API support for direct object file generation, eliminating the need for external `llc` tool

## Overview

This session completed the LLVM C API migration by implementing direct object file generation using the TargetMachine API. This eliminates the dependency on the external `llc` tool and makes the bootstrap compiler more self-contained. We also identified the need for `define-bitcode-ffi-function` to support declaring external functions from Vibe code.

## Problem Statement

The test script was using `llc` from PATH (version 6.0.0) while bitcode was generated by LLVM 21.1.6, causing version mismatch errors. The solution was to use LLVM C API's `LLVMTargetMachineEmitToFile()` to compile bitcode directly to object files, eliminating the need for `llc` entirely.

## Implementation Steps

### Step 1: Add TargetMachine FFI Functions

**File**: `bootstrap/runtime/ffi.ll`

Added FFI wrappers for LLVM TargetMachine API:
- `llvm_get_default_target_triple()` - Get default target triple
- `llvm_get_target_from_triple()` - Get target from triple string  
- `llvm_create_target_machine()` - Create target machine
- `llvm_target_machine_emit_to_file()` - Emit module to object file
- `llvm_dispose_target_machine()` - Clean up target machine

**Types added**:
- `%LLVMTargetRef = type i8*`
- `%LLVMTargetMachineRef = type i8*`

### Step 2: Add printf Declaration via LLVM API

**File**: `bootstrap/compiler/codegen.ll`

Modified `codegen_init()` to add printf declaration to the LLVM module via API:
- Created function type: `i32 (i8*, ...)` (vararg function)
- Added function declaration to module with name "printf"
- **Key fix**: Removed incorrect linkage setting - external functions default to ExternalLinkage (0), and explicitly setting it was causing verification errors

### Step 3: Add Function Lookup FFI Functions

**File**: `bootstrap/runtime/ffi.ll`

Added:
- `llvm_get_named_function()` - Look up functions by name
- `llvm_get_named_global()` - Look up global variables by name

### Step 4: Migrate codegen_call to Use LLVM API

**File**: `bootstrap/compiler/codegen.ll`

Modified `codegen_call()` to generate LLVM call instructions via builder:
- Looks up function in module by name using `llvm_get_named_function()`
- Creates argument values from AST nodes
- Generates call instruction using `llvm_build_call()`
- Falls back to text IR generation if LLVM API unavailable

**Note**: Currently falls back to text IR when functions aren't found (e.g., `define-bitcode-function` functions aren't parsed into module yet).

### Step 5: Ensure codegen_main Uses LLVM API

**File**: `bootstrap/compiler/codegen.ll`

Fixed `codegen_main()` to use LLVM API path:
- Creates main function structure via LLVM API
- Creates builder and positions it at entry block
- Generates code for top-level expressions (currently falls back to text IR for calls)
- Generates return instruction via LLVM API

### Step 6: Add Data Layout Support

**File**: `bootstrap/runtime/ffi.ll`, `bootstrap/compiler/codegen.ll`

Added `llvm_set_data_layout()` function to set data layout on module:
- Required for proper code generation
- Set to `"e-m:o-i64:64-f80:128-n8:16:32:64-S128"` (matching LLVM IR target datalayout)

### Step 7: Add Direct Object File Generation

**File**: `bootstrap/compiler/codegen.ll`

Added `codegen_write_object_file()` function:
- Gets target from triple
- Creates target machine with default options
- Emits module to object file using `llvm_target_machine_emit_to_file()`
- Returns 0 on success, -1 on error

### Step 8: Update main.ll for Object File Output

**File**: `bootstrap/compiler/main.ll`

Added support for detecting `.o` extension:
- Added `check_extension()` helper function
- When `.o` extension detected, calls `codegen_write_object_file()` instead of `codegen_write_bitcode()`
- Maintains backward compatibility with `.bc` extension

### Step 9: Update Test Script

**File**: `test/run_test.sh`

Updated to use object file output directly:
- Removed `llc` step entirely
- Compiler invocation outputs `.o` file directly
- Links executable directly from object file

### Step 10: Add Native Target Initialization (Critical Fix)

**File**: `bootstrap/runtime/ffi.ll`, `CMakeLists.txt`

**Problem**: TargetMachine API requires native target initialization before use. The initialization functions (`LLVMInitializeNativeTarget`, etc.) are `static inline` macros that expand to target-specific functions.

**Solution**: Added FFI declarations for X86-specific initialization functions:
- `LLVMInitializeX86TargetInfo()`
- `LLVMInitializeX86Target()`
- `LLVMInitializeX86TargetMC()`
- `LLVMInitializeX86AsmPrinter()`
- `LLVMInitializeX86AsmParser()`

**CMakeLists.txt updates**:
- Added `native` and `nativecodegen` components to LLVM_LIBS
- These components provide the X86 initialization functions

**Implementation**:
- Created `llvm_initialize_native_target()` wrapper function
- Calls all X86 initialization functions
- Called from `llvm_ffi_init()` to ensure initialization happens before any TargetMachine API usage

## Key Technical Insights

### Linkage for External Functions

External function declarations (like `printf`) should **not** have linkage explicitly set. The default ExternalLinkage (0) is correct. Setting it to 1 (which we thought was ExternalLinkage) actually caused module verification errors: "Global is external, but doesn't have external or weak linkage!"

### Native Target Initialization

The `LLVMInitializeNativeTarget()` functions are `static inline` macros that expand to target-specific functions. For x86_64, they expand to:
- `LLVMInitializeX86TargetInfo()`
- `LLVMInitializeX86Target()`
- `LLVMInitializeX86TargetMC()`
- `LLVMInitializeX86AsmPrinter()`
- `LLVMInitializeX86AsmParser()`

These must be called before using TargetMachine API, and they're in the `native`/`nativecodegen` LLVM components.

### PHI Node Placement

Fixed a critical bug where PHI nodes were not at the start of a basic block. PHI nodes must be the first instructions in a basic block in LLVM IR.

## Plan for `define-bitcode-ffi-function`

When we begin the process of rewriting all `.ll` files in `.vibe`, we should implement `define-bitcode-ffi-function` to:

1. **Declare external functions from Vibe code**:
   ```scheme
   (define-bitcode-ffi-function printf
     (return-type |i32|)
     (params (|i8*| format) ...)
     (linkage external)
     (vararg #t))
   ```

2. **Replace hardcoded declarations**: Move `printf` and other C library function declarations from hardcoded `codegen_init()` calls to Vibe code.

3. **Support user-defined FFI**: Allow users to declare their own external functions for calling C libraries.

4. **Potential use for initialization**: Could potentially use `define-bitcode-ffi-function` for target initialization functions, though they're currently called in `llvm_ffi_init()` which is fine for bootstrap infrastructure.
